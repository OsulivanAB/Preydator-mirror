### Trap Minimap Marker Feasibility Research (Issue #8)

## Request

Add the ability to mark prey trap locations on minimap/map.

## API findings (March 5, 2026)

### Relevant APIs

1. `C_VignetteInfo.GetVignettes()`
   - Returns currently known vignette GUIDs.
2. `C_VignetteInfo.GetVignetteInfo(vignetteGUID)`
   - Returns vignette metadata including `name`, `vignetteID`, and minimap/world-map presence fields.
3. `C_VignetteInfo.GetVignettePosition(vignetteGUID, uiMapID)`
   - Returns map position vector for a vignette.
4. `VIGNETTES_UPDATED`
   - Event for vignette list changes.
5. `NAME_PLATE_UNIT_ADDED`
   - Event when a nameplate unit appears; useful as a nearby detection signal.
6. `C_QuestLog.GetNextWaypoint(questID)`
   - Provides only the next objective waypoint, not guaranteed per-trap positions.

## Feasibility conclusion

- **Possible if traps are exposed as vignettes or nameplated units.**
- **Not possible** to mark hidden trap objects if Blizzard does not expose them through addon APIs.

## Recommended implementation approach

1. Add a debug mode to capture nearby candidate vignette/unit data in prey context.
2. Build a whitelist matcher (by `vignetteID`, name pattern, or GUID traits) for confirmed trap signals.
3. Render custom minimap/world map pins only for matched traps.
4. Add options:
   - Enable trap markers
   - Marker size/icon style
   - Show on minimap, world map, or both

## Validation checklist

1. Enter prey zone with debug mode enabled.
2. Trigger or approach known traps.
3. Record vignette and nameplate data snapshots.
4. Confirm repeatable identifiers across sessions.
5. Verify markers disappear when trap no longer exists.

## Risks

1. Trap mechanics may be intentionally hidden from addon APIs.
2. Name-based matching may break with localization.
3. Overly broad matching may mark unrelated world objects.

## Initial prototype scope

- Start read-only: detection + debug print, no permanent pins yet.
- After identifiers are confirmed, add pin rendering and player-facing options.
