### Stage Cleanup and Progress Mode Option (Issue #5)

## Problem statement

Current behavior effectively skips Stage 1 as soon as the correct zone is entered, which makes Stage 1 logic confusing and redundant.

## Decisions captured

1. Remove current Stage 1 behavior and renumber cleanly.
2. Move toward a modular addon structure instead of single-file architecture.
3. Preserve existing player settings during migration.
4. Add a player setting to choose progress mode:
   - Quarters (`25/50/75/100`)
   - Thirds (`33/66/100`)

## Functional direction

- Stage model:
  - Internal stage progression should match what players actually experience.
  - Legacy stage values should be migrated to new stage keys when needed.
- Progress mode option:
  - Add a dropdown in options: `Progress Segments` with `Quarters` and `Thirds`.
  - Tick marks, labels, and fallback percent logic should respect selected mode.

## Migration requirements

- Keep `PreydatorDB` backward compatible.
- On load, run a migration pass that:
  - Detects older stage label/sound key layouts.
  - Remaps stage keys to the new numbering.
  - Sets a default `progressSegments = "quarters"` for existing installs.
- Avoid resetting player-selected colors/fonts/sound paths.

## Proposed modular split

1. `core/state.lua` for runtime state and update loop.
2. `core/progress.lua` for percent extraction and stage mapping.
3. `ui/bar.lua` for bar creation, visuals, and animation.
4. `ui/options.lua` for settings UI controls.
5. `audio/sounds.lua` for sound path resolution and playback.
6. `data/defaults.lua` for constants/default settings.
7. `migration/schema.lua` for DB migrations.

## Acceptance criteria

1. Existing users keep their settings after update.
2. Stage display no longer includes the redundant old Stage 1 behavior.
3. Players can switch thirds/quarters in options without reload.
4. Tick marks and percent fallback align with selected mode.
5. No regression in sound playback or bar visibility logic.
