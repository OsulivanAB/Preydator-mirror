### Ambush Detection and Alert Direction (Issue #3)

## Decisions captured

1. `CHAT_MSG_SYSTEM` is no longer the primary plan.
2. Primary detection should be "rare scanner style" world detection.
3. Keep separate options for sound and visual alert behavior.
4. Default ambush response should be both sound and visual.
5. Do not inject additional chat output for ambush alerts.

## Detection strategy

### Primary path: vignette/nameplate style detection

- Evaluate detection via `C_VignetteInfo` and related events (`VIGNETTES_UPDATED`) to detect relevant world entities.
- Evaluate `NAME_PLATE_UNIT_ADDED` for nearby units tied to the ambush mechanic.
- Confirm if prey ambush entities expose stable identifiers (vignette ID, name, object GUID patterns, or unit name).

### Fallback path: pattern matching and debug capture

- If primary detection cannot reliably identify ambushes, add a pattern-matching fallback.
- Add a debug capture utility that logs candidate strings/events and makes copy/paste easy for field testing.
- Keep this tooling behind a debug option to avoid normal-play chat spam.

## Alert behavior

- Sound alert:
    - Reuse Stage 1 sound slot as the ambush sound trigger point.
    - Keep per-alert sound toggle separate from visual toggle.
- Visual alert:
    - Add a stylized `AMBUSH` state on the bar.
    - Add subtle shake animation while in combat.
    - Return bar to normal after combat ends or ambush state clears.
- Chat alert:
    - Not planned for default implementation.
